## [Unreleased]

### Changed
- Tuned the Bedrock + Azure OpenAI monitors to stop Slack alert flapping (all 23 orgs). The `@channel` handle was outside the `{{#is_alert}}` block, so every Warn/Recovered/Triggered transition pinged Slack; moved it into `{{#is_alert}}` + `{{#is_alert_recovery}}` so only critical alerts and their recoveries page (warnings stay on the dashboards). Also: rewrote `bedrock_server_errors` as an error **rate** (>10% of invocations over 15m) instead of a raw count (>5 in 5m) — a 0.6% transient blip no longer pages like a real 5.8%+ degradation; refit `bedrock_invocation_latency_high` to 45s (from 30s) for the heavier opus-4-8/4-7 model mix; added recovery-threshold hysteresis; dropped the `warning=1` on throttles; `renotify_interval` 30→60m on the three Bedrock metric monitors (the two Azure log alerts have no renotify).

### Removed
- `bedrock_invocations_drop` throughput-collapse monitor (all 7 orgs). Anomaly detection can't work on sparse/intermittent model traffic — it wedged in Alert on a model that stopped emitting and re-paged hourly. Redundant with `bedrock_invocation_latency_high`.

### Added
- Deployments & Rollouts dashboard (per tenant): every aspect of a deploy on one board, organized into collapsible sections — Pod age & rollout churn (avg pod age with the 90m/2h `pod_restart_storm` markers + a youngest-pods-in-minutes toplist color-coded to the storm thresholds + active-ReplicaSet count, the Problem 2 fleet-wide re-roll signature), Replica scaling (HPA current/desired vs min/max bounds), Autoscaler drivers (memory utilization % vs the 80% HPA target — the Problem 1 thrash driver, with the gsai-flux-mono PR #370 1000→1500Mi context), Rollout progress & availability (desired-vs-updated + unavailable replicas + cumulative restarts), and Deploy events (Kubernetes/Flux rollout timeline + stream). Namespace-scoped via a `$kube_namespace` template variable wired into every widget. All signals probed live against doc kube-state-metrics before building. Motivated by GSA-TTS/usai#1095.
- Pod-restart-storm monitor (`query alert`, all tenants): fires when a deployment's pods **never age past 90 minutes across a 4-hour window** (`max(last_4h)` of `avg:kubernetes_state.pod.age`, grouped by cluster+namespace+deployment) — i.e. pods are being replaced faster than they can age while the image never changes. Sourced from `pod.age` deliberately, NOT `container.restarts`: fresh pods start at 0 restarts, so a restart-count monitor is blind to this (which is why it went unalerted). `deployment_unavailable` also misses it — churning pods keep reaching Ready between restarts. Uses `max` (not `avg`) over the window: a one-off — or even 1–2× — rollout lets its new pods age monotonically past 2h, whereas a real storm pins the peak near the churn interval (~1h). Empirically verified: real sss/chat storm peaked ~3.9k s over 4h vs a Problem-2 flux rollout's ≥8.1k s floor — the 5,400s (90m) threshold sits inside that gap. Because a single synchronized rollout can no longer trip any group, this also removes the cross-org email flood at its source (there is no cross-org notification rollup — each tenant is an isolated Datadog org). Wide recovery hysteresis (critical 90m → recover 3h). Motivated by a 2026-07-22 storm on doc/ftc core deployments (`chat` cycled ~348 pods/6h on doc, 178 on ftc, vs ~8 on healthy gsa), driven by an external-secrets refresh → reloader restart loop; retuned 2026-07-23 after the original `avg`-based query paged all ~38 deployments across all 25 orgs (~150 emails) for a single fleet-wide gsai-flux-mono rollout.
- CronJob-failure monitor (`query alert`, all orgs): pages when a scheduled job's runs keep failing — `sum(last_1h) kubernetes_state.job.completion.failed by {cluster,namespace,cronjob}.as_count() >= 3` (warn at 1), grouped so the alert names the offending CronJob, with recovery hysteresis and a 2h renotify. Deliberately counts failed runs rather than keying off `cronjob.last_successful_time`/`next_schedule_time`, which are NOT populated in these clusters' kube-state-metrics (verified empty on doc/gsa) — a "time since last success" monitor would silently never fire. Motivated by GSA-TTS/usai#1094 — doc's `console-data` CronJob hit `DeadlineExceeded` on every run for 14+ days (stalling in its S3 log-redaction step, spiking to 742 failed jobs/day) with zero alerting, found only by reading events by hand. Same silent-failure class as GSA-TTS/usai#896.
- aigov Keycloak monitors + dashboard brought under Terraform (ROADMAP P2). Wired an `aws.aigov`/`datadog.aigov` provider pair in `tenants.tf` and activated the previously-parked `monitors.tf` + `dashboard.tf` (renamed from `.aigov-pending`). aigov is the shared account: it gets ONLY the 5 Keycloak log-alert monitors and the Keycloak dashboard — no `model_backend_monitors` module. The pre-existing hand-created resources were adopted via `terraform import` (monitors 568525–568532, dashboard `g2g-uxq-vqh`), so no duplicates were created. Folded in the dashboard CPU/mem/pods tag fix (`app:keycloak` → `kube_deployment:keycloak`, which was live but not in source). Two known-benign perpetual `plan` diffs remain (no real drift): Datadog auto-attaches a "Datadog Runbook" `assets` block to each monitor after apply, and the dashboard round-trips `notify_list = null`. The dashboard JSON carries no `tags` — the aigov org restricts dashboard tag keys to `team`/`ai` and rejects `managed-by:terraform`/`service:keycloak` (those stay on the monitors). This closes the drift gap that let the dashboard widgets silently break on the wrong tag.
- Deployment-availability monitor (`query alert`, all 23 orgs): fires when a deployment is short of its desired replicas for a sustained 30m (`kubernetes_state.deployment.replicas_desired − replicas_ready >= 1`, grouped by namespace+deployment). A normal rolling-update dip recovers within the window; `on_missing_data=default` avoids false pages on intentional teardown. Paired with a "Deployment Health" section on the Infra Health dashboard (unavailable-replicas bars + top-offenders toplist). Motivated by GSA-TTS/usai#896 — a legacy frontend-apps deployment on doc/dot/usda ran a vulnerable image and sat "does not have minimum availability" for a full day with zero alerting.
